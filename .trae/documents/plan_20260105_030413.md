## 修复应用闪退问题的详细计划

### 第一步：修改 SuperpoweredModule.java（Java 层）

**1.1 动态获取音频参数**
- 使用 AudioManager 获取设备真实的采样率 (sampleRate)
- 使用 AudioManager 获取设备推荐的缓冲区大小 (bufferSize)
- 将这些参数传递给 C++ 层初始化函数

**1.2 修改 initialize() 方法**
```java
// 动态获取音频参数
int sampleRate = AudioManager.getProperty(AudioManager.PROPERTY_OUTPUT_SAMPLE_RATE);
int bufferSize = AudioManager.getProperty(AudioManager.PROPERTY_OUTPUT_FRAMES_PER_BUFFER);
// 如果获取失败，使用默认值
if (sampleRate == 0) sampleRate = 44100;
if (bufferSize == 0) bufferSize = 512;
// 传递给 C++ 层
nativeInitialize(sampleRate, bufferSize);
```

**1.3 创建新的 startAudio() JNI 方法**
- 将 audioIO->start() 调用从 nativeInitialize 移到 nativeStartAudio()
- 确保音频启动只在用户点击播放按钮时执行

### 第二步：修改 ResonareAudioEngine.cpp（C++ 层）

**2.1 修改 nativeInitialize 函数签名**
- 添加 sampleRate 和 bufferSize 参数
- 使用传入的参数初始化音频 IO，而非硬编码值

**2.2 添加 nativeStartAudio 函数**
```cpp
extern "C" JNIEXPORT void JNICALL
Java_com_resonare_SuperpoweredModule_nativeStartAudio(JNIEnv *env, jobject __unused obj) {
    if (audioIO == nullptr) {
        LOGE("Cannot start audio - audioIO is null");
        return;
    }
    audioIO->start();
    isPlaying = true;
    LOGD("Audio started successfully");
}
```

**2.3 添加全面的防御性代码**
- 在所有函数开头添加 nullptr 检查
- 添加 try-catch 块处理异常
- 记录详细的错误日志（包含时间戳、错误类型和上下文）
- 移除所有可能导致断言失败的代码

**2.4 简化 audioProcessing 回调**
- 暂时输出静音数据，确保不崩溃
- 后续再逐步恢复音频生成逻辑

### 第三步：测试验证

**3.2 运行测试**

- 切换不同场景，确认功能正常
- 长时间运行，确认不会闪退

**3.3 异常场景测试**
- 模拟内存不足情况

- 确认应用只输出错误日志而不会闪退
